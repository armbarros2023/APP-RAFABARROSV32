Manual do Sistema & Especificação Técnica

Sistema: Equipe Rafael Barros V31.0
Objetivo: Guia de uso e especificação para migração backend.

1. Visão Geral

Aplicação Single Page (SPA) em React com TypeScript. Atualmente utiliza localStorage para persistência de dados (Client-Side). O objetivo da migração para o backend é centralizar estes dados em um banco SQL/NoSQL e prover uma API segura.

2. Estrutura de Dados (Schema)

As seguintes chaves são utilizadas no armazenamento local e representam as tabelas necessárias no banco de dados:

Entidade / Chave Descrição Relacionamentos
equipe_rafael_barros_students Cadastro de alunos, dados financeiros e responsáveis. 1:N com Agendamentos, N:1 com Filial e Terapeuta
equipe_rafael_barros_team Cadastro de colaboradores (terapeutas e staff). N:1 com Filial
equipe_rafael_barros_users_db Credenciais de acesso (Login). Vinculado ao StaffMember
equipe_rafael_barros_branches Filiais da clínica (Multi-tenancy). Entidade Raiz

3. Pontos de Atenção para o Backend

Segurança: A API Key do Google Gemini está exposta no frontend atualmente. No backend, ela deve ser movida para variáveis de ambiente do servidor e acessada via Proxy/Endpoint dedicado.
  • Autenticação: Implementar JWT (JSON Web Tokens). O frontend já possui um AuthContext preparado para receber tokens.
  • Uploads: Implementar rota de upload de arquivos (S3/Blob Storage) para substituir os URLs locais simulados (URL.createObjectURL).
  • Agendamento: A lógica de conflito de horários deve ser validada no servidor para evitar race conditions.

4. Guia de Funcionalidades (Testes)

  • Dashboard: Verifique se os totais financeiros batem com a soma das transações no banco.
  • Financeiro: Teste a criação de uma fatura e a mudança de status para "Pago". Verifique se o recibo PDF é gerado com os dados corretos.
  • IA: Teste a geração de relatórios na aba do aluno. O backend deve receber o histórico em texto e devolver a análise do Gemini.
